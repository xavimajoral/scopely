## 1. .NET 8.0 and C# Basics

### What is .NET?

.NET is a free, cross-platform, open-source developer platform for building many types of applications. The backend uses **.NET 8.0**, the latest LTS (Long Term Support) version.

### Key C# Language Features Used

#### 1.1 Namespaces
Namespaces organize code into logical groups and prevent naming conflicts.

```csharp
namespace SupportTicketingSystem.Domain;
```

**In the codebase:**
- `SupportTicketingSystem.Domain` - Domain entities (Ticket, Reply)
- `SupportTicketingSystem.Data` - Data access layer (DbContext, Repositories)
- `SupportTicketingSystem.Services` - Business logic layer
- `SupportTicketingSystem.Api` - API controllers and DTOs

#### 1.2 Classes and Properties

```csharp
public class Ticket
{
    public int Id { get; set; }
    public string Subject { get; set; } = string.Empty;
    public DateTime CreatedAt { get; set; } = DateTime.UtcNow;
}
```

- `{ get; set; }` - Auto-implemented properties
- `= string.Empty` - Default value initialization
- `DateTime.UtcNow` - Gets current UTC time

#### 1.3 Enums

```csharp
public enum TicketStatus
{
    Open = 0,
    InResolution = 1,
    Resolved = 2
}
```

Enums represent a set of named constants. Used for ticket status in the domain model.

#### 1.4 Nullable Reference Types

```csharp
<TargetFramework>net8.0</TargetFramework>
<Nullable>enable</Nullable>
```

When enabled, reference types are non-nullable by default. Use `?` to make them nullable:
- `Ticket?` - Can be null
- `Ticket` - Cannot be null (compiler warning if null)

#### 1.5 Implicit Usings

```csharp
<ImplicitUsings>enable</ImplicitUsings>
```

Automatically imports common namespaces, so you don't need `using System;`, `using System.Linq;`, etc.

#### 1.6 File-Scoped Namespaces

```csharp
namespace SupportTicketingSystem.Domain;

public class Ticket { }
```

Newer C# feature - namespace doesn't require braces when it spans the entire file.

## 2. Project Structure and Solution Organization

### Layered Architecture (Clean Architecture)

The backend uses a **layered architecture** with clear separation of concerns:

```
SupportTicketingSystem.sln (Solution)
├── SupportTicketingSystem.Api (Presentation Layer)
├── SupportTicketingSystem.Services (Business Logic Layer)
├── SupportTicketingSystem.Data (Data Access Layer)
└── SupportTicketingSystem.Domain (Domain Layer)
```

### Dependency Flow (One-Way)

```
Api → Services → Data → Domain
```

- **Domain**: Core business entities (no dependencies)
- **Data**: Implements data access (depends on Domain)
- **Services**: Business logic (depends on Data, Domain)
- **Api**: Controllers and HTTP handling (depends on Services, Domain)

**Example from `SupportTicketingSystem.Api.csproj`:**
```xml
<ProjectReference Include="..\SupportTicketingSystem.Domain\SupportTicketingSystem.Domain.csproj" />
<ProjectReference Include="..\SupportTicketingSystem.Services\SupportTicketingSystem.Services.csproj" />
<ProjectReference Include="..\SupportTicketingSystem.Data\SupportTicketingSystem.Data.csproj" />
```

### Project Types

- **Library Projects** (`.csproj`): Domain, Data, Services
- **Web Project** (`Microsoft.NET.Sdk.Web`): Api project

## 3. ASP.NET Core Web API

ASP.NET Core is a cross-platform framework for building web APIs and web apps.

### 3.1 Program.cs - Application Entry Point

In .NET 6+, `Program.cs` replaces `Startup.cs`. This is where you configure your application.

**Key Components:**

```csharp
var builder = WebApplication.CreateBuilder(args);
```

Creates a builder to configure services and the app.

```csharp
var app = builder.Build();
```

Builds the application.

```csharp
app.Run();
```

Starts the web server.

### 3.2 Middleware Pipeline

The request pipeline processes HTTP requests:

```csharp
app.UseSwagger();           // Swagger documentation (dev only)
app.UseHttpsRedirection();  // Redirect HTTP to HTTPS (prod)
app.UseCors("AllowFrontend"); // Enable CORS
app.MapControllers();        // Map controller routes
```

**Order matters!** Middleware executes in the order it's registered.

## 4. Dependency Injection (DI)

Dependency Injection is a design pattern where objects receive their dependencies from an external source rather than creating them.

### 4.1 Service Registration

In `Program.cs`:

```csharp
// Register DbContext
builder.Services.AddDbContext<ApplicationDbContext>(options =>
    options.UseSqlite($"Data Source={dbPath}"));

// Register repositories
builder.Services.AddScoped<ITicketRepository, TicketRepository>();
builder.Services.AddScoped<IReplyRepository, ReplyRepository>();

// Register services
builder.Services.AddScoped<ITicketService, TicketService>();
```

### 4.2 Service Lifetimes

1. **Scoped** (`AddScoped`): Created once per HTTP request
   - Used for: Repositories, Services, DbContext
   
2. **Singleton** (`AddSingleton`): Created once for the application lifetime
   - Used for: Configuration, logging

3. **Transient** (`AddTransient`): Created every time they're requested
   - Used for: Lightweight, stateless services

**Why Scoped for Repositories?**
- Each HTTP request gets its own DbContext instance
- Ensures thread-safety and proper disposal

### 4.3 Constructor Injection

Dependencies are injected through constructors:

```csharp
public class TicketsController : ControllerBase
{
    private readonly ITicketService _ticketService;
    private readonly ILogger<TicketsController> _logger;

    public TicketsController(ITicketService ticketService, ILogger<TicketsController> logger)
    {
        _ticketService = ticketService;
        _logger = logger;
    }
}
```

The DI container automatically provides these dependencies when creating the controller.

### 4.4 Interface-Based Design

Using interfaces (`ITicketService`) instead of concrete classes (`TicketService`) enables:
- **Testability**: Easy to mock interfaces
- **Flexibility**: Swap implementations without changing dependent code
- **Dependency Inversion Principle**: Depend on abstractions, not concretions

## 5. Entity Framework Core (EF Core)

EF Core is an Object-Relational Mapping (ORM) framework that lets you work with databases using .NET objects.

### 5.1 DbContext

`ApplicationDbContext` is the bridge between your domain entities and the database:

```csharp
public class ApplicationDbContext : DbContext
{
    public DbSet<Ticket> Tickets { get; set; }
    public DbSet<Reply> Replies { get; set; }
}
```

- `DbSet<T>` represents a table in the database
- Methods like `Add()`, `Update()`, `Remove()` track changes
- `SaveChangesAsync()` writes changes to the database

### 5.2 Configuration (OnModelCreating)

```csharp
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    modelBuilder.Entity<Ticket>(entity =>
    {
        entity.HasKey(e => e.Id);
        entity.Property(e => e.Subject).IsRequired().HasMaxLength(200);
        
        entity.HasMany(e => e.Replies)
              .WithOne(e => e.Ticket)
              .HasForeignKey(e => e.TicketId)
              .OnDelete(DeleteBehavior.Cascade);
    });
}
```

This method configures:
- **Primary Keys**: `HasKey()`
- **Required Fields**: `IsRequired()`
- **Max Length**: `HasMaxLength()`
- **Relationships**: `HasMany().WithOne()` (One-to-Many)
- **Cascade Delete**: When a ticket is deleted, its replies are deleted too

### 5.3 Relationships

**One-to-Many**: One Ticket has many Replies

```csharp
// In Ticket class
public ICollection<Reply> Replies { get; set; } = new List<Reply>();

// In Reply class
public int TicketId { get; set; }
public Ticket Ticket { get; set; } = null!;
```

- `TicketId` - Foreign key (stored in database)
- `Ticket` - Navigation property (used for loading related data)

### 5.4 SQLite Database

```csharp
builder.Services.AddDbContext<ApplicationDbContext>(options =>
    options.UseSqlite($"Data Source={dbPath}"));
```

SQLite is a file-based database - perfect for development and small applications. The database file is `support_tickets.db`.

### 5.5 Database Initialization

```csharp
using (var scope = app.Services.CreateScope())
{
    var context = scope.ServiceProvider.GetRequiredService<ApplicationDbContext>();
    context.Database.EnsureCreated();
}
```

`EnsureCreated()` creates the database if it doesn't exist. **Note**: This is for development. Production should use migrations.

## 6. Repository Pattern

The Repository Pattern abstracts data access logic, making the code more testable and maintainable.

### 6.1 Interface Definition

```csharp
public interface ITicketRepository
{
    Task<IEnumerable<Ticket>> GetUnresolvedTicketsAsync();
    Task<Ticket?> GetTicketByIdAsync(int id);
    Task<Ticket> CreateTicketAsync(Ticket ticket);
    Task<Ticket> UpdateTicketAsync(Ticket ticket);
}
```

The interface defines **what** operations are available, not **how** they're implemented.

### 6.2 Implementation

```csharp
public class TicketRepository : ITicketRepository
{
    private readonly ApplicationDbContext _context;

    public TicketRepository(ApplicationDbContext context)
    {
        _context = context;
    }

    public async Task<IEnumerable<Ticket>> GetUnresolvedTicketsAsync()
    {
        return await _context.Tickets
            .Where(t => t.Status != TicketStatus.Resolved)
            .OrderByDescending(t => t.CreatedAt)
            .ToListAsync();
    }
}
```

**Key Points:**
- Uses `ApplicationDbContext` to access data
- Methods are `async` (return `Task` or `Task<T>`)
- Uses LINQ for queries
- `ToListAsync()` executes the query asynchronously

### 6.3 Benefits

1. **Separation of Concerns**: Controllers don't know about EF Core
2. **Testability**: Easy to create mock repositories
3. **Flexibility**: Can swap data access technology (EF Core → Dapper → Raw SQL)
4. **Single Responsibility**: Repository handles only data access

## 7. Service Layer Pattern

The Service Layer contains business logic and orchestrates operations between repositories.

### 7.1 Service Interface

```csharp
public interface ITicketService
{
    Task<Ticket> CreateTicketAsync(string subject, string description, string username, string userId);
    Task<Ticket> ResolveTicketAsync(int ticketId);
    Task<Reply> AddReplyAsync(int ticketId, string message, string username, string userId, bool isFromAgent);
}
```

Services expose business operations, not data access operations.

### 7.2 Service Implementation

```csharp
public class TicketService : ITicketService
{
    private readonly ITicketRepository _ticketRepository;
    private readonly IReplyRepository _replyRepository;

    public TicketService(ITicketRepository ticketRepository, IReplyRepository replyRepository)
    {
        _ticketRepository = ticketRepository;
        _replyRepository = replyRepository;
    }

    public async Task<Reply> AddReplyAsync(int ticketId, string message, string username, string userId, bool isFromAgent)
    {
        var ticket = await _ticketRepository.GetTicketByIdAsync(ticketId);
        if (ticket == null)
        {
            throw new ArgumentException($"Ticket with ID {ticketId} not found", nameof(ticketId));
        }

        // Business rule: If an agent replies, update ticket status to InResolution
        if (isFromAgent && ticket.Status == TicketStatus.Open)
        {
            ticket.Status = TicketStatus.InResolution;
            ticket.UpdatedAt = DateTime.UtcNow;
            await _ticketRepository.UpdateTicketAsync(ticket);
        }

        var reply = new Reply { /* ... */ };
        return await _replyRepository.CreateReplyAsync(reply);
    }
}
```

**Key Points:**
- Contains **business logic** (e.g., "If agent replies, change status")
- Uses multiple repositories if needed
- Validates business rules
- Throws exceptions for invalid operations

### 7.3 Why Separate Service Layer?

1. **Business Logic**: Rules belong in services, not controllers
2. **Reusability**: Services can be used by controllers, background jobs, etc.
3. **Testability**: Easy to unit test business logic
4. **Transaction Management**: Can coordinate multiple repository operations

## 8. DTOs (Data Transfer Objects)

DTOs are simple objects used to transfer data between layers (API ↔ Domain).

### 8.1 Why DTOs?

- **Security**: Don't expose internal domain properties
- **API Design**: Control what clients send/receive
- **Versioning**: Change DTOs without changing domain models
- **Validation**: Add validation attributes to DTOs

### 8.2 Input DTOs (CreateTicketDto)

```csharp
public class CreateTicketDto
{
    [Required]
    [StringLength(200)]
    public string Subject { get; set; } = string.Empty;

    [Required]
    public string Description { get; set; } = string.Empty;
}
```

Used for data coming **into** the API (HTTP POST/PUT).

### 8.3 Output DTOs (TicketDto)

```csharp
public class TicketDto
{
    public int Id { get; set; }
    public string Subject { get; set; } = string.Empty;
    public TicketStatus Status { get; set; }
    public List<ReplyDto> Replies { get; set; } = new();
}
```

Used for data going **out** of the API (HTTP GET responses).

### 8.4 Mapping

Controllers map between Domain entities and DTOs:

```csharp
private static TicketDto MapToDto(Ticket ticket)
{
    return new TicketDto
    {
        Id = ticket.Id,
        Subject = ticket.Subject,
        Status = ticket.Status,
        Replies = ticket.Replies.Select(MapToDto).ToList()
    };
}
```

**Note**: In larger projects, use AutoMapper or Mapster for mapping.

## 9. Controllers and Routing

Controllers handle HTTP requests and return HTTP responses.

### 9.1 Controller Setup

```csharp
[ApiController]
[Route("api/[controller]")]
public class TicketsController : ControllerBase
{
}
```

- `[ApiController]` - Enables API-specific features (automatic model validation, etc.)
- `[Route("api/[controller]")]` - Route template (`[controller]` = "Tickets")
- `ControllerBase` - Base class for API controllers (not `Controller`, which is for MVC views)

### 9.2 HTTP Methods

```csharp
[HttpGet]                    // GET /api/tickets
public async Task<ActionResult<IEnumerable<TicketDto>>> GetUnresolvedTickets()

[HttpGet("{id}")]            // GET /api/tickets/5
public async Task<ActionResult<TicketDto>> GetTicket(int id)

[HttpPost]                   // POST /api/tickets
public async Task<ActionResult<TicketDto>> CreateTicket([FromBody] CreateTicketDto dto)

[HttpPost("{id}/resolve")]   // POST /api/tickets/5/resolve
public async Task<ActionResult<TicketDto>> ResolveTicket(int id)
```

**Route Parameters:**
- `{id}` - Captures URL segment as parameter
- `[FromBody]` - Deserializes JSON request body into DTO

### 9.3 Return Types

```csharp
Task<ActionResult<TicketDto>>
```

- `ActionResult<T>` - Can return multiple HTTP status codes
- `Ok(ticketDto)` - 200 OK
- `NotFound("message")` - 404 Not Found
- `CreatedAtAction(...)` - 201 Created with Location header
- `StatusCode(500, "message")` - 500 Internal Server Error

### 9.4 Error Handling

```csharp
try
{
    var ticket = await _ticketService.GetTicketByIdAsync(id);
    if (ticket == null)
    {
        return NotFound($"Ticket with ID {id} not found");
    }
    return Ok(MapToDto(ticket));
}
catch (Exception ex)
{
    _logger.LogError(ex, "Error getting ticket {TicketId}", id);
    return StatusCode(500, "An error occurred while retrieving the ticket");
}
```

- `try-catch` handles exceptions
- `_logger.LogError()` logs errors
- Returns appropriate HTTP status codes

## 10. Async/Await Programming

Asynchronous programming prevents threads from blocking while waiting for I/O operations (database, HTTP calls, file system).

### 10.1 Async Methods

```csharp
public async Task<Ticket?> GetTicketByIdAsync(int id)
{
    return await _context.Tickets
        .FirstOrDefaultAsync(t => t.Id == id);
}
```

- `async` - Method can use `await`
- `Task<T>` - Represents an asynchronous operation that returns `T`
- `Task` - Represents an operation that returns nothing
- `await` - Waits for async operation to complete

### 10.2 Why Async?

**Synchronous (Blocking):**
```csharp
var tickets = _context.Tickets.ToList(); // Thread waits here
```

**Asynchronous (Non-Blocking):**
```csharp
var tickets = await _context.Tickets.ToListAsync(); // Thread can do other work
```

Benefits:
- Better scalability (handles more concurrent requests)
- Better resource utilization
- UI stays responsive

### 10.3 Async All The Way

If a method is `async`, all methods it calls should be `async` too:

```
Controller (async) → Service (async) → Repository (async) → EF Core (async)
```

**Anti-pattern:**
```csharp
// DON'T DO THIS
public async Task<Ticket> GetTicket(int id)
{
    return _repository.GetTicket(id); // Missing await!
}
```

**Correct:**
```csharp
public async Task<Ticket?> GetTicket(int id)
{
    return await _repository.GetTicketAsync(id);
}
```

## 11. LINQ (Language Integrated Query)

LINQ allows you to query collections using C# syntax.

### 11.1 Query Syntax vs Method Syntax

**Method Syntax (Used in codebase):**
```csharp
var unresolved = await _context.Tickets
    .Where(t => t.Status != TicketStatus.Resolved)
    .OrderByDescending(t => t.CreatedAt)
    .ToListAsync();
```

**Query Syntax (Alternative):**
```csharp
var unresolved = from t in _context.Tickets
                 where t.Status != TicketStatus.Resolved
                 orderby t.CreatedAt descending
                 select t;
```

### 11.2 Common LINQ Methods

- **`.Where()`** - Filters elements
  ```csharp
  .Where(t => t.Status != TicketStatus.Resolved)
  ```

- **`.OrderBy()` / `.OrderByDescending()`** - Sorts elements
  ```csharp
  .OrderByDescending(t => t.CreatedAt)
  ```

- **`.FirstOrDefaultAsync()`** - Gets first element or null
  ```csharp
  .FirstOrDefaultAsync(t => t.Id == id)
  ```

- **`.Include()`** - Loads related entities (eager loading)
  ```csharp
  .Include(t => t.Replies.OrderBy(r => r.CreatedAt))
  ```

- **`.Select()`** - Transforms elements
  ```csharp
  tickets.Select(MapToDto).ToList()
  ```

- **`.ToListAsync()`** - Executes query and materializes results

### 11.3 Deferred Execution

LINQ queries are **deferred** - they don't execute until you enumerate them:

```csharp
var query = _context.Tickets.Where(t => t.Status == TicketStatus.Open);
// Query not executed yet!

var tickets = await query.ToListAsync();
// Query executes here
```

This allows building queries dynamically.

## 13. CORS (Cross-Origin Resource Sharing)

CORS allows web pages to make requests to a different domain than the one serving the web page.

### 13.1 CORS Configuration

```csharp
builder.Services.AddCors(options =>
{
    options.AddPolicy("AllowFrontend", policy =>
    {
        policy.WithOrigins(allowedOrigins.ToArray())
              .AllowAnyMethod()
              .WithHeaders("Content-Type", "Authorization")
              .AllowCredentials();
    });
});

app.UseCors("AllowFrontend");
```

**What it does:**
- `WithOrigins()` - Allows specific frontend URLs (localhost:5173, etc.)
- `AllowAnyMethod()` - Allows GET, POST, PUT, DELETE, etc.
- `WithHeaders()` - Allows specific HTTP headers
- `AllowCredentials()` - Allows cookies/auth headers

**Why needed?**
- Frontend (localhost:5173) makes requests to API (localhost:5000)
- Browser blocks cross-origin requests by default
- CORS tells browser it's OK

## 15. Data Validation

### 15.1 Data Annotations

```csharp
public class CreateTicketDto
{
    [Required]
    [StringLength(200)]
    public string Subject { get; set; } = string.Empty;
}
```

**Common Attributes:**
- `[Required]` - Field must be provided
- `[StringLength(200)]` - Maximum string length
- `[Range(1, 100)]` - Numeric range
- `[EmailAddress]` - Valid email format
- `[RegularExpression]` - Custom regex pattern

### 15.2 Automatic Validation

With `[ApiController]`, validation happens automatically:

```csharp
[HttpPost]
public async Task<ActionResult<TicketDto>> CreateTicket([FromBody] CreateTicketDto dto)
{
    // If dto is invalid, returns 400 Bad Request automatically
    // No need to check ModelState.IsValid
}
```

Invalid requests return `400 Bad Request` with validation errors.

### 15.3 Custom Naming Policy

```csharp
public class UpperSnakeCaseNamingPolicy : JsonNamingPolicy
{
    public override string ConvertName(string name)
    {
        var result = Regex.Replace(name, "(?<!^)([A-Z])", "_$1");
        return result.ToUpperInvariant();
    }
}

builder.Services.AddControllers()
    .AddJsonOptions(options =>
    {
        options.JsonSerializerOptions.Converters.Add(
            new JsonStringEnumConverter(namingPolicy: new UpperSnakeCaseNamingPolicy()));
    });
```

Converts enum values: `TicketStatus.Open` → `"OPEN"` in JSON.

## Key Patterns Summary

### Layered Architecture
- **Domain**: Entities, enums (no dependencies)
- **Data**: DbContext, Repositories (depends on Domain)
- **Services**: Business logic (depends on Data, Domain)
- **Api**: Controllers, DTOs (depends on Services, Domain)

### Dependency Injection
- Register services in `Program.cs`
- Inject via constructors
- Use interfaces, not concrete classes

### Repository Pattern
- Abstracts data access
- Makes code testable
- Separates concerns

### Service Layer
- Contains business logic
- Orchestrates repositories
- Validates business rules

### Async/Await
- Use `async Task<T>` for async methods
- Use `await` for async calls
- Async all the way through the stack

## Codebase-Specific Notes

### Project Dependencies

```
SupportTicketingSystem.Api
├── Microsoft.AspNetCore.OpenApi (8.0.22)
├── Swashbuckle.AspNetCore (6.6.2) - Swagger
└── References: Domain, Services, Data

SupportTicketingSystem.Data
├── Microsoft.EntityFrameworkCore.Sqlite (8.0.0)
├── Microsoft.EntityFrameworkCore.Design (8.0.0)
└── References: Domain

SupportTicketingSystem.Domain
└── No dependencies (pure domain layer)

SupportTicketingSystem.Services
└── References: Domain, Data (via repository interfaces)
```

### Database File Location
- Development: `support_tickets.db` in API project folder
- Render (production): Uses `RENDER_DISK_PATH` environment variable

### Port Configuration
- Development: Uses `launchSettings.json`
- Render: Uses `PORT` environment variable
